# Messenger — Overview

## 1. Описание проекта

Данный проект представляет собой распределённую систему мессенджера, построенную на микросервисной архитектуре.

Система позволяет пользователям:
- создавать личные и групповые чаты;
- отправлять текстовые сообщения;
- получать историю переписки;
- редактировать и удалять сообщения;
- получать сообщения в реальном времени;
- видеть статусы доставки и прочтения сообщений;
- видеть online/offline статус других пользователей.

Архитектура основана на подходе **Event-Driven Architecture (EDA)** с использованием Kafka для обмена событиями между сервисами.

---

## 2. Основные сервисы

Система состоит из трёх основных микросервисов:

### 2.1 Chat Service
Отвечает за управление чатами и участниками.

Функции:
- создание личных и групповых чатов;
- управление участниками (добавление/удаление);
- хранение информации о чатах;
- проверка членства пользователя в чате.

Хранилище:
- PostgreSQL

---

### 2.2 Message Service
Отвечает за работу с сообщениями.

Функции:
- отправка сообщений;
- хранение истории переписки;
- редактирование и удаление сообщений;
- выдача истории сообщений;
- хранение статусов доставки и прочтения;
- публикация событий о сообщениях.

Хранилище:
- PostgreSQL

---

### 2.3 Notification / Realtime Gateway
Отвечает за доставку сообщений в реальном времени.

Функции:
- управление websocket-соединениями;
- отправка событий клиентам в реальном времени;
- отслеживание online/offline статуса пользователей;
- обработка delivery/read подтверждений;
- маршрутизация соединений.

Хранилище:
- Redis (ephemeral state)

---

## 3. Общая архитектура

Система состоит из следующих компонентов:

- Client (Web/Mobile)
- API Gateway (опционально)
- Chat Service
- Message Service
- Realtime Gateway
- PostgreSQL (для хранения данных)
- Redis (для realtime состояния)
- Kafka (для обмена событиями)

### Упрощённая схема:
Client
↓
API Gateway
↓
Message Service → PostgreSQL
↓
Kafka
↓
Realtime Gateway → WebSocket → Client

Chat Service → PostgreSQL
↓
Kafka

## 4. Основные сценарии работы

### 4.1 Отправка сообщения

1. Клиент отправляет HTTP-запрос в Message Service  
2. Message Service проверяет, состоит ли пользователь в чате (через Chat Service)  
3. Сообщение сохраняется в PostgreSQL  
4. Message Service публикует событие `message.created` в Kafka  
5. Realtime Gateway получает событие  
6. Gateway отправляет сообщение пользователям через WebSocket  

---

### 4.2 Получение истории сообщений

1. Клиент отправляет запрос в Message Service  
2. Message Service проверяет доступ  
3. Message Service читает данные из PostgreSQL  
4. Возвращает список сообщений  

---

### 4.3 Realtime подключение

1. Клиент устанавливает WebSocket соединение с Realtime Gateway  
2. Gateway аутентифицирует пользователя  
3. Gateway сохраняет соединение в Redis  
4. Пользователь считается online  

---

### 4.4 Обработка read/delivered

1. Клиент отправляет событие `message.read` или `message.delivered`  
2. Gateway публикует событие `receipt.updated` в Kafka  
3. Message Service сохраняет статус в базе  

---

## 5. Хранилища данных

Система использует несколько типов хранилищ:

### PostgreSQL
Используется для:
- чатов;
- участников;
- сообщений;
- версий сообщений;
- статусов доставки и прочтения.

---

### Kafka
Используется для:
- передачи событий между сервисами;
- асинхронной обработки;
- обеспечения слабой связанности сервисов.

Kafka **не используется как основное хранилище данных**.

---

### Redis
Используется для:
- хранения online/offline статуса пользователей;
- хранения websocket-соединений;
- routing;
- временных данных (TTL).

---

## 6. Основные принципы архитектуры

### 6.1 Разделение ответственности
Каждый сервис отвечает за свою область:
- Chat → структура общения  
- Message → содержимое общения  
- Realtime → доставка  

---

### 6.2 Event-Driven взаимодействие
Сервисы взаимодействуют через Kafka:
- сервис публикует события;
- другие сервисы подписываются на них.

---

### 6.3 Источник истины
- сообщения хранятся в Message Service (PostgreSQL);
- Kafka используется только для событий;
- Redis — для временного состояния.

---

### 6.4 Масштабируемость
- все сервисы масштабируются горизонтально;
- Kafka позволяет выдерживать пики нагрузки;
- Redis обеспечивает быстрый доступ к realtime данным.

---

## 7. Scope MVP

В рамках MVP реализуется:

- создание чатов;
- отправка сообщений;
- получение истории;
- редактирование/удаление сообщений;
- realtime доставка сообщений;
- статус online/offline;
- delivery/read подтверждения.

---

## 8. Вне scope MVP

Не входят в MVP:
- вложения (media);
- push-уведомления;
- сложный поиск (Elasticsearch);
- реакции и треды;
- end-to-end шифрование;
- multi-device sync сложного уровня.
